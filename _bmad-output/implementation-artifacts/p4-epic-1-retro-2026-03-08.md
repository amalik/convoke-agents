## Successes

- **Scope was well-sized** — both stories shipped cleanly with zero regressions

## Challenges

- **Initial review was skimmed** — Story 1.1's first review pass marked everything "N/A" without verification; user had to push back to get a genuine review. Second pass found 4 real issues.
- **False task completion** — Story 1.1 Task 1.6 ("Verify on GitHub") was marked [x] when changes hadn't been committed or pushed yet
- **Incomplete discovery during story creation** — Story 1.2 scope was based on searching 1 file, not the codebase. A simple grep during story creation would have found all 37 occurrences.
- **Dev Notes not followed** — File List used line number references despite explicit prohibition in Dev Notes

## Key Insights

1. **Discovery gaps get caught in review, but should be caught in story creation** — a codebase-wide grep during story creation prevents scope surprises
2. **Review quality requires accountability** — lazy first passes happen; the system needs the user to push back
3. **Skipping stages is fine when proportional** — quick wins don't need PRDs, but logic changes (Epic 3) need readiness gates
4. **Task completion criteria must be verified literally** — "Verify on GitHub" means actually verify on GitHub

## Action Items

### Process Improvements

1. **Do codebase-wide discovery during story creation, not review**
   - Apply to: All stories with search-and-replace or pattern-fix scope
   - Success criteria: Story file includes a verified occurrence inventory before marking `ready-for-dev`

2. **Match process weight to risk level**
   - Quick wins / visual polish: lightweight (initiative → epic → stories → implement)
   - Logic changes / system internals: full gates (readiness check, architecture review)
   - Apply to: Epic 3 onward

3. **Verify task completion criteria literally before marking [x]**
   - "Verify on GitHub" means actually verify on GitHub
   - Apply to: All stories

### Team Agreements

- Review passes must be genuine — no blanket "N/A" without evidence
- Visual stories (Epic 2) require actual verification notes in completion record
- File List entries use content anchors, not line numbers
