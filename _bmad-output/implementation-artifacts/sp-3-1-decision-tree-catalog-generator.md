# Story SP-3.1: Decision-Tree Catalog Generator

Status: ready-for-dev

## Story

As a consultant new to agentic tools,
I want a browsable catalog README organized by what I'm trying to do (intent categories),
so that I can find the right skill in under 60 seconds without knowing BMAD internals.

## Acceptance Criteria

1. A catalog generator script exists at `scripts/portability/catalog-generator.js` (CommonJS, executable via `node`). It reads `_bmad/_config/skill-manifest.csv` and `_bmad/_config/agent-manifest.csv`, and produces a `README.md` with a decision-tree structure grouped by intent category. The script is pure-output (reads manifests, writes one file) and has no side effects on the source tree.

2. **Intent category headings** are phrased as "I need to..." prompts that match the vision doc's catalog framing:
   - `think-through-problem` → "I need to think through a problem"
   - `define-what-to-build` → "I need to define what to build"
   - `review-something` → "I need to review something"
   - `write-documentation` → "I need to write documentation"
   - `plan-your-work` → "I need to plan my work"
   - `test-your-code` → "I need to test my code"
   - `discover-product-fit` → "I need to discover product-market fit" (Tier 3 — pipeline only, shown with note)
   - `assess-readiness` → "I need to assess production readiness" (Tier 3 — pipeline only, shown with note)
   - `meta-platform` → excluded from the catalog (framework-internal skills, not user-facing)

   The mapping is a static lookup table inside the script — NOT derived from the intent string dynamically. This ensures wording stability across regenerations. Unknown intents fall back to `"I need to: <intent>"` with a warning on stderr.

3. **Each skill entry** within an intent section shows, on a single line or compact block:
   - Persona icon (from agent-manifest `icon` column, or `🔧` for workflow-derived personas)
   - Persona name (from agent-manifest `displayName`, or humanized skill name for workflow-derived)
   - One-line description (from skill-manifest `description` column, first sentence only — truncate at first period + space or at 120 chars, whichever comes first)
   - Tier badge: `[Ready to use]` for standalone, `[Needs setup]` for light-deps, `[Framework only]` for pipeline
   - Link to skill folder: `[→ instructions](./skill-name/)` (relative link assumes the catalog README lives at the repo root alongside the skill directories, per the vision doc structure)

   Skills are sorted alphabetically within each intent section. Duplicate skill names (same skill across modules) appear only once — reuse the `Set`-based dedup pattern from sp-2-3/sp-2-4.

4. **Standalone skills are visually distinguished.** The `[Ready to use]` badge is rendered as a bold green marker: `**✅ Ready to use**`. Light-deps skills show `**📦 Needs setup**`. Pipeline skills show `**🔒 Framework only**` followed by "(Requires full Convoke installation)".

5. **The catalog README includes a header block** with:
   - Title: `# Convoke Skills Catalog`
   - Subtitle: `> Find the right AI skill for your task. Browse by what you're trying to do.`
   - A "How to use a skill" quick-start section (3 lines: 1. Find your intent, 2. Pick a skill marked "Ready to use", 3. Copy the folder into your project's `.claude/skills/` directory)
   - A legend explaining the three tier badges
   - Skill count split by visibility: `**{M} skills in this catalog** ({S} ready to use, {L} need setup) | {P} framework-only skills listed below`. Where M = S + L (main-body skills only — Tier 1 + Tier 2), and P = Tier 3 count (collapsed section). This avoids the confusing "80 skills available" headline when only 49 are visible above the fold.

6. **The catalog README includes a footer** with:
   - "Generated by `convoke-export` — do not edit manually" note
   - Generation timestamp (ISO 8601 date, no time — e.g., `2026-04-10`)
   - Link to the main Convoke repo for the full framework

7. **CLI interface** matches the existing portability scripts pattern:
   - `node scripts/portability/catalog-generator.js` — writes to stdout (default)
   - `node scripts/portability/catalog-generator.js --output <path>` — writes to file at `<path>`
   - `node scripts/portability/catalog-generator.js --help` — prints usage and exits 0
   - Exit code 0 on success, 1 on usage error, 2 on manifest read failure
   - No `--dry-run` needed (the script is inherently read-only on the source tree; it either writes stdout or a single file)

8. **Persona resolution for the catalog** reuses the sp-2-2/sp-2-4 engine's persona stack — but the catalog generator does NOT call `exportSkill()` (too expensive for a catalog). Instead, it uses a lightweight path:
   - Read agent-manifest.csv and build a `skillName → { displayName, icon }` lookup using the same Strategy 1 + Strategy 2 + Strategy 2b matching logic from `loadPersona()`.
   - For skills that don't match any agent-manifest row, fall back to `{ displayName: humanizeSkillName(name), icon: '🔧' }` (same as Strategy 5's output shape).
   - Extract the matching logic into a new exported helper `resolvePersonaSummary(skillName, agents)` in `export-engine.js` that returns `{ name, icon }` without reading any source files. This avoids duplicating the Strategy 1/2/2b matching logic.
   - **Scope of the shared `findAgentMatch` internal function:** Strategies 1 + 2 + 2b ONLY (manifest-based matching, no file reads). Strategy 3 (description fuzzy match — reads skill content for "talk to X" patterns) stays exclusively in `loadPersona()` because it requires file I/O. Strategy 4 (inline extraction) and Strategy 5 (workflow-derived) also stay in `loadPersona()` only. `resolvePersonaSummary` calls `findAgentMatch` and on miss returns the `humanizeSkillName` + `🔧` fallback directly — it does NOT attempt Strategies 3-5.

9. **A test file at `tests/lib/portability-catalog-generator.test.js`** adds at least 6 tests:
   - **Test 1: stdout mode produces valid markdown.** Run the generator with no flags, capture stdout. Assert it starts with `# Convoke Skills Catalog`, contains at least 4 intent section headings (the 6 standalone intents minus the 2 pipeline-only ones that may or may not appear), and contains the persona name `Carson`.
   - **Test 2: --output mode writes file.** Run with `--output <tmpfile>`, assert file exists, is non-empty, and contains the `# Convoke Skills Catalog` title heading. Do NOT assert byte-for-byte match with stdout mode — the footer's generation timestamp could differ if the two runs straddle a date boundary.
   - **Test 3: all 6 standalone intent categories present.** Assert the output contains all 6 "I need to..." headings for the standalone intents.
   - **Test 4: tier badges present.** Assert output contains `Ready to use` (from the 44 standalone skills in the main body) AND `Framework only` (from the 36 pipeline skills in the collapsed `<details>` section per AC #10). `Needs setup` may or may not appear depending on whether light-deps skills are present — assert it only if the manifest has light-deps skills.
   - **Test 5: skill count line matches manifest.** Parse the `**M skills in this catalog**` line. Assert M equals the unique Tier 1 + Tier 2 (non-meta-platform) skill count from the manifest (currently 44 standalone + 5 light-deps = 49, minus any meta-platform overlap). The P count (framework-only) should equal the unique Tier 3 non-meta-platform count.
   - **Test 6: no BMAD internals leak.** Assert the output does NOT contain `_bmad/`, `bmad-init`, `.claude/hooks`, `{project-root}`, or any of the FORBIDDEN_STRINGS from sp-2-1.

10. **The generator only includes Tier 1 and Tier 2 skills in the main catalog body.** Tier 3 (pipeline) skills are listed in a collapsed `<details>` section at the bottom titled "Framework-only skills (requires Convoke installation)" so they're discoverable but not prominent. `meta-platform` intent skills are excluded entirely (they're framework internals, not user-facing).

## Dev Notes

### Architecture Compliance (CRITICAL)

- **Read-only on source tree.** The generator reads manifests and optionally writes ONE output file. It does NOT modify any source files.
- **CommonJS.** Match existing convention.
- **No new npm dependencies.** Only `fs`, `path`, and existing project utilities.
- **`findProjectRoot()` from `scripts/update/lib/utils.js`** for manifest paths.
- **Reuse `readManifest` from `manifest-csv.js`** and `resolvePersonaSummary` from `export-engine.js`.
- **The generator does NOT call `exportSkill()`.** That's too expensive (reads all source files per skill). The catalog only needs manifest metadata + persona name/icon — `resolvePersonaSummary()` provides that without touching the filesystem beyond the two CSV files.

### Carson duplicate: `bmad-brainstorming` and `bmad-cis-agent-brainstorming-coach`

Both are standalone, both resolve to Carson 🧠, both are in `think-through-problem`. They are distinct skills with different instructions (one is the user-facing skill, the other is the CIS agent definition). The catalog will show both — this is correct and intentional. Do NOT dedupe by persona name; dedupe only by skill name (which is already unique after `Set`-based dedup). The reader sees two Carson entries with different descriptions and can pick either.

### The catalog's audience

The catalog README will live at the root of the `convoke-skills-catalog` repo (sp-4-1 creates and seeds it). Its readers are **consultants who don't know BMAD** — they browse GitHub, see skill folders, and need to understand what each one does and whether they can use it directly. The "I need to..." decision-tree structure is the core UX: a reader with a task in mind scrolls to their intent section and picks a skill.

### Skill description truncation

Manifest descriptions are verbose ("Use when the user asks to talk to X or requests the Y"). The catalog shows only the first sentence. Truncate at `description.indexOf('. ') + 1` (first period + space). If no period, take the first 120 chars and append `...`. This keeps entries scannable.

### Duplicate skill names

The manifest has duplicate rows (same skill name across modules). The generator uses `Set`-based dedup (same pattern as sp-2-3/sp-2-4) so each skill appears once in the catalog. The first row wins for metadata (description, tier, intent).
