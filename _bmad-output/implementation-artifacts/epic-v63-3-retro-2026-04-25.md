## What Didn't Go Well

### 1. V-pass methodology has a subtle pitfall: shell pipeline `$?` reads tail's exit
Discovered at Story 3.5 dev-time as AC2 deviation: V-pass empirical probe claimed `validate-exports` exits 0 even with issues. Reality: validator exits 1 on any issue. The probe shell pipeline used `... 2>&1 | tail -30; echo "exit=$?"` — `$?` read `tail`'s exit, not `node`'s. Substantive criterion (allowlist semantics) still met; spec amended inline. **Process fix:** future V-pass probes that capture exit codes MUST use `${PIPESTATUS[0]}` (bash) or `set -o pipefail` or run the command without pipes when exit is the load-bearing signal.

### 2. Story 3.4 Task 0 baseline FAILED at 8 pre-existing tests
Story 3.1's skill-dir migration (`<id>.md` → `<id>/SKILL.md`) didn't update integration test fixtures referencing the old layout. Discovered at Story 3.4 dev-story Task 0.1 baseline gate. Pattern: **structural migrations need explicit fixture-update tasks** in their own story spec — not assumed. Story 3.1 V-pass + R1 + R2 reviews didn't catch this because the affected tests pass `runScript()` with `cwd: PACKAGE_ROOT`-substitute fixtures; the broken fixture-state was only exposed when Story 3.4 ran the integration suite as its baseline gate. Operator-managed-side-commit (BUG-6) cleaned it up; story 3.4 resumed cleanly. Process improvement: structural migrations should add a "downstream-test-fixture audit" subtask to AC or a deferred Bug-Lane row at story-author time.

### 3. Story 3.5 spot-check rubric was too narrow
The 9-cell rubric (frontmatter shape + `## You are` + `FORBIDDEN_STRINGS`) PASSED 9/9 — and missed `[your name]`/`[your context]`/`[your preferred language]`/`[your output folder]` template placeholders leaked across **44/44 exported instructions.md**. Caught only by Edge Case Hunter's broader sweep. Pattern: spec spot-check rubrics should be designed to catch the most user-visible content defects, not just structural shape. Routed to deferred-work as Bug-Lane (export-engine fix) + Initiative-Lane (rubric-widening). RICE candidate flagged.

### 4. AC4 catalog walkthrough N/A pattern was a spec-design slip on Story 3.5
The spec assumed `validate-exports` would auto-generate Manual Catalog Walkthrough that applied to raw export output — but those checks gate on the catalog README that only `seed-catalog-repo.js` produces. Story 3.5 Task 4 had to mark 4 catalog walkthrough boxes `[N/A — out of Story 3.5 AC3 scope]`. Auditor judged the N/A defensible (catalog README doesn't exist in raw export); but the spec design slipped on this initially. Pattern: when a spec consumes an existing tool with a multi-section report template, audit ALL sections not just the load-bearing ones — N/A scope should be authored, not discovered.

### 5. Sprint-status.yaml `last_updated` field continues to bloat
Each story close adds a multi-thousand-character paragraph as the "current" `last_updated` value, with prior summaries appended via `| Prior:` pipes. After 5 stories + multiple V-passes + R1+R2 cycles, the field is ~50KB of prose in a single YAML scalar. Pre-existing pattern; flagged in 3.5 R1 as R1-M6 (deferred). Future fix needed: trim `last_updated` to date + 1-line pointer; move detail to a separate changelog field or rely on per-story Change Log tables.

### 6. Cron-schedule durability claim unreliable
3 crons scheduled this session (Story 3.3 PR check + Story 3.4 BMAD-source check + Story 3.5 portability re-verification) all returned "Session-only (not written to disk, dies when Claude exits)" despite passing `durable: true`. The claimed-durable mechanism is not honored by the runtime. Workaround: re-schedule from a fresh session approaching firing date. **Documentation gap** — the cron tool description says `durable: true` persists, but the runtime contradicts. Worth filing upstream.

## Key Lessons (bite-sized, for reuse)

1. **Empirical probe before spec-rewrite.** When V-pass critical findings cluster around tool-contract assumptions, RUN the tools first. Reality grounds ACs better than inference.
2. **Edge Case Hunter is the load-bearing reviewer for procedural stories.** Independent re-run via Bash catches what spec-shape-checking misses.
3. **DEF-SPIKE inversion → spec-rewrite (option 4) is faster than patching.** When a load-bearing Decision is invalidated, halt + present options + spec-rewrite. Don't try to retrofit.
4. **Operator-managed-side-commit keeps story branches scope-pure.** When discovered work is off-scope (BUG-6, R2-A1, A39, A41/A42), HALT + surface + operator commits separately. Story diff stays clean.
5. **Convergence rule applies cleanly to procedural stories.** Story 3.5 closed at R1 with 0 HIGH after triage — first v6.3 story to do so. Pattern: clear empirical baseline + cross-layer verification = faster convergence.
6. **Procedural stories have a documented shape now.** Spec + 1-2 evidence reports (frontmatter + per-cell tables + structured findings + path-safety analysis for tear-downs). 3 exemplars in v6.3.
7. **Shell pipeline `$?` reads the LAST command's exit, not the first.** Use `${PIPESTATUS[0]}`, `set -o pipefail`, or no pipes when capturing exit codes empirically.
8. **Spec spot-check rubrics need to catch user-visible defects, not just shape.** Story 3.5 R1-M4 (template placeholder leak) was a spec-rubric design slip — caught only by Edge Case Hunter's broader sweep. Apply rubric-broadening at spec-author time.
9. **Structural migrations need explicit downstream-test-fixture audits.** Story 3.1's skill-dir migration left 8 broken integration tests; only discovered at Story 3.4 baseline gate. Pattern: structural migrations should add a fixture-update subtask or deferred Bug-Lane row at story-author time.
10. **`durable: true` cron flag may not be honored** — runtime returns session-only. Re-schedule from fresh session if firing date crosses session boundary.
