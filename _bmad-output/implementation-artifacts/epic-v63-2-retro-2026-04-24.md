## What Went Well

### 1. The convergence rule held — even in Round 2

Every story with HIGH findings in R1 triggered R2 automatically. Two stories (2.2, 2.3) converged cleanly at R2 with zero HIGH carrying forward. Story 2.4 was the interesting case: R2 itself surfaced **3 new HIGH findings** (formula-sanitization breaking R1-M4 verification, SharedArrayBuffer + unguarded unlink, symlink traversal) — but because all R2 patches were **non-structural** (logic tweaks within existing functions, not new files or renamed functions), the rule said "converge at R2" and held. The rule's "structural changes" discriminator proved useful: it distinguished "we found more bugs, fix them here" from "we need a third look."

### 2. Upstream API reuse discipline was tight

Zero reimplementation across the four stories. Story 2.2's doctor imports `scanBmmDependencies` / `readExistingCsv` / `mergePreservingManual` directly from 2.1. Story 2.3's gate imports `checkBmmDependencies` from 2.2 via lazy-require. Story 2.4's CLI imports `_tripleKey` / `_atomicWrite` / `_inferSourceModule` / `renderCsv` / `CSV_HEADER_FIELDS` / `OUTPUT_CSV_REL` — six separate upstream consumers from a single `require('./audit/audit-bmm-dependencies')`. The review agents explicitly verified these imports against the upstream export block in every round.

**Anti-pattern held:** `_sanitizeFormula` was an exception — needed at the verifyRegistration layer (Story 2.4 R2-H1) but not exported from upstream. Decision at dev time: mirror locally with a `_sanitizeForCompare` helper + an explicit "kept in lock-step with upstream" JSDoc comment, rather than modify upstream (out of scope). This was the right call for 2.4's scope; a future sprint could promote `_sanitizeFormula` to the upstream `_internal` if other consumers emerge.

### 3. Decisions resolved cleanly at spec time — not patched in code review

**Story 2.1 D1** (prefix-rule extension): resolved at validate-create-story. **Story 2.2 D1** (`softWarning` field vs. separate `warning` type vs. exit-code logic): resolved at validate-create-story. **Story 2.3 D1** (gate on both `refresh-only` + `upgrade` vs. one path): resolved at validate-create-story — critically, V-pass caught that the initial spec only wired the `upgrade` path. **Story 2.4 D1** (interactive vs. non-interactive vs. both): resolved at spec time via the "slash-command LLM conversation wraps CLI" insight. Across all four stories, only **1 decision ended up in R2** (2.1 R2-D3 triple-key dedup shape). This is the validate-create-story ROI — catching decisions before dev saves Round 1 theater.

### 4. Pattern reuse accelerated each successive story

- **Atomic write wrapped in advisory lockfile** (new in 2.4 R1-H1): `fs.openSync(path, 'wx')` + retry-with-spin. Pairs with the existing `_atomicWrite` (from 1A.4 → 2.1) but adds multi-process safety. Re-usable in any future scenario where two CLI invocations might race a shared file.
- **`softWarning` field + exit-code filter** (from 2.2): `checks.filter(c => !c.passed && !c.softWarning)` is now a 1-liner for "hard fails only" across any health check. Reused by 2.3's gate + 2.4's verification.
- **Lazy-require inside try/catch** (from 2.3 R1-M2): downstream module loads only when needed; a failure surfaces as a fail-soft warning instead of aborting startup. Reused in 2.4's `verifyRegistration` call path.
- **`Object.freeze(_internal)` export block** (from 2.3 R2-L5): prevents test-order leaks via mutated internals. Reused in 2.4. This small hardening turned out to be important: Story 2.4's test suite was the largest yet and would have been vulnerable without it.
- **Slash-command skill wrapper pattern** (new in 2.4): `.claude/skills/bmad-*/SKILL.md` + `workflow.md` (single file, 4 phases) for conversational LLM layer on top of a tested CLI. The `bmad-*` skill folder + `convoke-*` CLI bin convention is now explicit.

### 5. `validate-create-story` paid for itself on every single story

Every V-pass found 6–18 improvements (avg ~10). Story 2.3's V-pass caught the missing refresh-only path in Decision 1 — that would have been a 2-round review issue to unearth. Story 2.4's V-pass caught 6 CRITICAL issues including the AC5 summary-mode breakage that would have shipped a silently-broken verification path. The pass has become non-optional in the operating pipeline.

### 6. Parallel 3-reviewer adversarial model keeps finding distinct things

Across 8 review rounds, Blind Hunter + Edge Case Hunter + Acceptance Auditor consistently surfaced non-overlapping findings. Blind finds the "how does this break under adversarial input" bugs; Edge walks the exhaustive boundary cases; Auditor holds the implementation honest against the spec. Story 2.4 R2 was a clean demonstration: all 3 HIGH findings came from different reviewers (R2-H1 from Blind+Edge, R2-H2 from Edge, R2-H3 from Edge). One-reviewer coverage would have shipped at least one HIGH.

## What Didn't Go Well

### 1. Test-fragility snuck through on multiple stories before Round 2 caught it

Story 2.3 Legacy Test 9 passed via incidental substring match in unrelated output (caught R2-M2, test deleted). Story 2.4 Test 12 SIGINT test passed via Node's default signal behavior, never exercising the custom handler (caught R2-H2, test repurposed as R1-M8 non-TTY guard). Story 2.4 R1-H1 concurrency test is STILL weak — two `setTimeout(fn, 0)` on a single-threaded event loop is not actually concurrent, but the lockfile is correct in principle so this was deferred rather than fixed (see R2-M1 defer).

**Root cause:** spawn-based tests with assertions keyed on substring presence are fragile. Shared fixture output (validator messages, scanner logs) can silently satisfy test assertions that were supposed to verify narrower things.

**Lesson:** for tests that verify a specific CLI-emitted string, prefer AND-conjunction of multiple specific substrings (as R2-M2 rewrote Test 2 of Story 2.3). For tests that verify a specific side effect (byte-identical file post-cancel, specific exit code), prefer asserting on that side effect rather than on a stdout substring.

### 2. LOC projections were off again — but the 60% overhead budget held

- **Story 2.1:** projected 250–350 → shipped 420 (+20%). Within budget.
- **Story 2.2:** projected 80–120 → shipped 178 (+48% on the midpoint). Within budget.
- **Story 2.3:** projected 40–60 → shipped 100 (+100% on midpoint). **Over** the naive budget but on a tiny base where small additions (JSDoc, defensive field guards, `_internal` block) multiply quickly.
- **Story 2.4:** projected 180–260 → shipped ~630 after R1+R2 (+160% on midpoint). **Significantly over** — driven by advisory lockfile (R1-H1), 3-layer path-traversal guard (R1-H4 + R2-H3), full-row verification (R1-M4 + R2-H1), argv parser error handling (R1-H3/M2/M3/M5), non-TTY guard (R1-M8). Each individually small; cumulatively large.

**Lesson:** the 60% overhead budget from Epic 1A PI-3 still applies to stories with 200+ LOC baseline. For <100-LOC stories (like 2.3's gate helper), absolute LOC of defensive hardening approaches the story size. **New heuristic:** baseline projection + ~40 LOC per HIGH finding + ~10 LOC per MED finding (retrospectively derived from 2.4's 4+3 HIGH count).

### 3. `validate-create-story` catches drift between spec body and post-R1 reality

Story 2.4 R2-D1: the spec AC8 body still listed 12 test cases when the code had 19. R2-D2: R1-M1 patch description said 3 prefixes; code shipped 4 (redundant-but-equivalent). These were caught at Round 2 (by the Acceptance Auditor), fixed in-round, but represent a structural drift problem: **once R1 patches land, the spec body becomes stale.** The Change Log + File List entries were accurate; the AC body wasn't.

**Lesson:** Round 1 / Round 2 patches that change test count or scope should trigger a spec-body update alongside the code change. Story 2.4's dev was the first to hit this because it had the most spec/code delta; 2.1/2.2/2.3 were small enough to stay aligned.

### 4. Story 2.4's R1-H3 over-rejection was a Round 1 patch that needed Round 2 refinement

The R1-H3 fix for "next-token-starts-with-dash" initially used `.startsWith('-')`, which wrongly blocked legitimate dash-leading values like `--email -alice@foo.com` or `--source -internal`. This was caught when the R2-H1 test for formula-sanitization couldn't pass: the email value `-alice@foo.com` was rejected by the parser before even reaching the sanitization logic. Narrowed R1-H3 to `.startsWith('--')` during R2 patch-apply.

**Lesson:** fixes for one class of bug (silently-consumed flag-shaped values) can over-catch a related class (legitimate dash-leading strings). Round 2 tests that exercise adjacent behavior catch this. The R2-H1 test was the accidental safety net.

### 5. The "workflow.md wraps tested CLI" pattern needs explicit operator-facing docs

Story 2.4 shipped a slash-command skill (`/bmad-register-skill`) that wraps the tested `convoke-register-skill` CLI. The workflow.md is LLM-readable prose, not operator-readable documentation. An operator typing `/bmad-register-skill` gets a guided flow via the LLM, but there's no plain-prose "here's what this does and when to use it" doc in `docs/` — the reader has to read workflow.md or the SKILL.md frontmatter trigger phrases.

**Deferred (R1-L7):** add an operator-facing affordance for `--dry-run` in workflow.md Phase 2. This is a symptom of a larger documentation gap: slash-command skills need a separate operator doc, not just an LLM-driving workflow.

### 6. Scope discipline held everywhere except one near-miss on Story 2.4

Story 2.4's R2-H1 fix (formula-sanitization parity) needed `_sanitizeFormula` — upstream-only. Option A was to modify `scripts/audit/audit-bmm-dependencies.js` (out of scope); Option B was to mirror the function locally. Went with B + a JSDoc comment "kept in lock-step with upstream." This works today but creates a documentation debt: **if upstream's `_sanitizeFormula` ever changes** (e.g., new sanitization sentinel character), the local copy drifts silently. No lint rule catches this.

**Lesson:** cross-file logic mirroring should emit a runtime assertion on load (e.g., hash the upstream function body and log a warning if it differs). Not doing this today; flagged as a future hardening.

## Action Items for Epic 3 (or next sprint)

**Process Improvements (carry-forward + new):**

1. **[PI-5 new] Spec-body drift check at Round 2.** When Round 1 patches change test count, AC scope, or task subtasks, update the spec body in the same R1 commit — don't wait for R2 Auditor to flag it. Owner: dev agent. Trigger: any R1 patch that modifies AC8's enumerated test list.

2. **[PI-6 new] Substring-assertion audit for CLI tests.** Tests that spawn a CLI and assert on stdout substring presence must: (a) use AND-conjunction of multiple specific substrings when matching a single finding, or (b) assert on a side effect (exit code, file state, byte-identity) rather than a rendered string. Applies to all future CLI-spawn tests.

3. **[PI-7 new] Non-TTY guard as template for operator-facing CLIs.** Story 2.4 R1-M8 (refuse-to-prompt-when-piped) should be the default for any new CLI with interactive prompts. Add to project-context.md anchor rules list.

4. **[PI-3 → carry] 60% overhead budget, refined.** Updated heuristic: baseline projection + ~40 LOC per anticipated HIGH finding + ~10 LOC per MED. Useful for story sizing when HIGH-finding risk is known in advance (e.g., path-handling code, concurrent file access, arg-parsing).

**Technical Debt:**

5. **[TI-4 new] Upstream `_sanitizeFormula` mirror drift.** `scripts/convoke-register-skill.js:_sanitizeForCompare` is a local copy of upstream logic. Either promote `_sanitizeFormula` to `scripts/audit/audit-bmm-dependencies.js` `_internal` exports (out-of-scope for 2.4, would need its own story), or add a runtime hash-check assertion that the two copies match at load time. Priority: low (upstream rule has been stable for months).

6. **[TI-5 new] `_withCsvLock` concurrency test is weak.** Story 2.4 R2-M1 defer: the R1-H1 concurrency test uses `Promise.all` + `setTimeout(fn, 0)` which is sequential on the event loop, not parallel. Real concurrency test would require spawning two subprocesses. Owner: whoever touches the lockfile next. Priority: low (lockfile is correct by construction).

7. **[TI-3 → carry] Pre-4.0 defer sweep.** Aggregate defer count across Epic 1A + Epic 2 is ~35 items. Most are low-impact UX polish; a few might be promotable to backlog rows. Suggest running `/bmad-enhance-initiatives-backlog triage` before the 4.0 release tag to audit the deferred-work.md file and decide what ships as known-issue vs. gets addressed.

**Documentation:**

8. **[DI-1 new] Slash-command skill operator docs.** Story 2.4's `/bmad-register-skill` ships with workflow.md (LLM-facing) and SKILL.md (trigger metadata). No operator-facing plain doc exists. For the Convoke 4.0 release, either `docs/bmad-register-skill.md` or a section in an operator-guide page is needed. Low urgency (LLM drives the UX today); revisit post-release if operators report confusion.
