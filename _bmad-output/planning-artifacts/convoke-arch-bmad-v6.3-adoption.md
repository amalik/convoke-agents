## Project Context Analysis

### Requirements Overview

**Functional Requirements: 50 FRs across 10 capability areas**

Architecturally, the FRs cluster into four distinct *system concerns* (not workstreams — the architecture cares about how things connect, not how work is sequenced):

| System Concern | FRs | Architectural shape |
|----------------|-----|---------------------|
| **Config loading redesign** | FR1–FR4 | Replace a skill-invocation pattern with a direct-YAML-read pattern across ~25 agent files. Affects the activation protocol for every Convoke agent. Single design decision, broad sweep. |
| **User migration pipeline** | FR5–FR11 | CLI script (`convoke-update`) that detects pre-v4 state, transforms it to v4 state, and validates the result. Idempotent, resume-safe, path-safe, offline-safe. Pipeline with well-defined stages. |
| **Governance & registry system** | FR12–FR18 | New subsystem: `bmm-dependencies.csv` registry generated by a scan tool, validated by `convoke-doctor`, gated by `convoke-update`. Three integration points into existing CLI infrastructure. |
| **Marketplace & distribution** | FR19–FR25 | Repo-level metadata files (`.claude-plugin/marketplace.json`, `module.yaml`) plus runtime compatibility preflight. Mostly static files with one dynamic check. Plus platform-agnostic adapter validation (EXP3 GO). |

Secondary concerns (lower architectural complexity):
- **Agent consolidation** (FR26–FR30): File removal + manifest updates. Mostly mechanical.
- **Release discipline artifacts** (FR31–FR35): Playbook file + ADR. Documentation, not code.
- **Validation battery** (FR36–FR40): New capability — test harness comparing pre/post agent outputs against drift threshold T.
- **Release communication** (FR41–FR44): Grep-testable CHANGELOG checks. Lightweight scripting.
- **Quality gates** (FR45–FR46): IR gate + architecture doc existence. Process, not infrastructure.
- **Retrospective artifacts** (FR47–FR50): Anti-pattern registry + findings pipeline. Documentation infrastructure.

**Non-Functional Requirements: 33 NFRs across 7 categories**

NFRs that **drive architectural decisions** (load-bearing):

| NFR cluster | What it demands architecturally |
|-------------|--------------------------------|
| **NFR6–NFR8 (Reliability)** | Migration script must be idempotent, resumable after interruption, and offline-safe. Requires *transactional state tracking* — manifest-based checkpoint approach, not naive file copying. |
| **NFR11–NFR15 (Integration)** | Convoke must conform to BMAD v6.3's PluginResolver, skill directory convention, activation protocol, and module.yaml schema. *External interface contracts* we don't control and must match exactly. |
| **NFR26–NFR29 (Backwards Compatibility)** | One-minor-version compat window. `bmad-init` deprecated in 4.0, warned on activation, removed in 4.1. Custom user skills must keep working. Requires *deprecation detection and warning* in the new direct-load path. |
| **NFR30–NFR33 (Reproducibility)** | Drift threshold T must be numeric, validation battery must be reproducible, same battery re-runnable post-release. Demands a *deterministic test harness* design. |

NFRs that are **constraints, not drivers** (shape solutions but don't create new architecture):
- Performance (NFR1–NFR5): Time bounds on CLI operations. Constrain implementation choices.
- Maintainability (NFR16–NFR20): Single-maintainer constraint. Affects complexity budget.
- Observability (NFR21–NFR25): Logging requirements. Affect output format, not system shape.

### Scale & Complexity

- **Primary domain:** Infrastructure/tooling for an AI agent framework (not web, mobile, or API)
- **Complexity level:** HIGH — concentrated in migration sweep breadth (~25 agents), marketplace integration conformance, governance registry design, and behavioral equivalence validation
- **System type:** Single-user CLI tool with filesystem-based state. No database, no auth, no network services, no concurrency. All operations are local, synchronous, and batch.
- **Estimated architectural components:** 6 distinct components

**What makes this architecturally unusual:** This is not a typical software architecture. The "architecture" is about:
1. **File layout conventions** — where things live, what patterns they follow
2. **Script design patterns** — how CLI tools compose, checkpoint, and validate
3. **Registry schemas** — CSV format for `bmm-dependencies.csv` and its scan/validate contract
4. **Validation protocols** — how PF1 battery works, what drift threshold T means operationally
5. **Integration conformance** — exact mapping to BMAD v6.3's PluginResolver expectations
6. **Backwards compatibility strategy** — deprecation detection, warning, and removal timeline

### Technical Constraints & Dependencies

**Host framework dependencies (first-class, from PRD classification):**
- `claude-code` — LLM runtime. Skill activation protocol, `.claude/skills/` directory convention, tool vocabulary.
- `bmad-method@6.3.0` — Convention framework. Config loading pattern, PluginResolver, marketplace registry schema, skill directory convention.
- `claude-opus-4-6` — Model behavior contract. Agent outputs depend on model behavior; drift in model = drift in agent outputs independent of Convoke changes.

**Existing infrastructure Convoke builds on:**
- `scripts/update/lib/utils.js` — `getPackageVersion()`, `compareVersions()`, `findProjectRoot()`
- `scripts/update/lib/refresh-installation.js` — shared refresh logic
- `scripts/update/lib/migration-runner.js` — orchestration for migration scripts
- `scripts/update/lib/config-merger.js` — `mergeConfig()` with YAML round-trip
- `scripts/update/lib/validator.js` — validates agents, workflows, config, manifest
- `scripts/update/migrations/registry.js` — append-only migration registry
- `scripts/portability/export-engine.js` — P18 skill exporter (confirmed via EXP3)
- `scripts/portability/generate-adapters.js` — platform adapter generation

**Constraints from Convoke's existing architecture rules:**
- NEVER hardcode version strings — read from `package.json` via `getPackageVersion()`
- NEVER use `process.cwd()` directly — use `findProjectRoot()` or accept `projectRoot` param
- Migration files = ONLY delta logic; `refreshInstallation()` handles file copying
- `_bmad/` directory paths are NOT renamed (BMAD Method compatibility)

### Cross-Cutting Concerns Identified

1. **Config loading pattern change affects ALL agents.** ~25-file sweep must be atomic in effect — partial migration leaves some agents on old pattern, others on new.
2. **Dual distribution must produce identical state.** npm install and marketplace install must yield same filesystem state. Any divergence = silent behavioral bug.
3. **Backwards compatibility window creates co-existence.** Between 4.0.0 and 4.1.0, direct-load utility must detect both patterns and warn on the old one — aware of the old pattern, not ignorant of it.
4. **Extensions governance is a new system boundary.** `bmm-dependencies.csv` is a new contract between Convoke and users' custom skills. Schema, scan tool, and validation gate must be designed as a coherent subsystem.
5. **Behavioral equivalence validation is a new capability.** PF1 battery requires representative inputs, pre/post agent execution, semantic diff within threshold T, and a go/no-go gate. Test harness doesn't exist today.
6. **Platform-agnostic adapter validation (post-EXP3).** WS2 includes verifying P18 exporter adapters across Claude Code, Copilot, and Cursor for representative Tier 1 skills.

### Decision 1: Config Loading Architecture (WR1 + WR8)

**Decision:** Create `scripts/update/lib/config-loader.js` as a shared utility module.

**API (revised per FM1-1 — handles nested module paths):**

```js
const { loadModuleConfig } = require('./config-loader');

// Accepts full relative module path (handles nested modules)
const vortexConfig = loadModuleConfig(projectRoot, 'bme/_vortex');
// Reads: {projectRoot}/_bmad/bme/_vortex/config.yaml

const coreConfig = loadModuleConfig(projectRoot, 'core');
// Reads: {projectRoot}/_bmad/core/config.yaml

const enhanceConfig = loadModuleConfig(projectRoot, 'bme/_enhance');
// Reads: {projectRoot}/_bmad/bme/_enhance/config.yaml
```

**Design details:**
- Single function, two parameters. `moduleConfigPath` is the relative path under `_bmad/`.
- Returns plain JS object from parsed YAML — same shape as what `bmad-init` returned.
- No caching, no singleton — each call reads fresh from disk.
- Uses `yaml` package already in Convoke's dependencies.
- Try/catch wraps YAML parse — on error: clear message naming file + line, never crash (FM1-3).

**Alternatives considered:**
1. *Inline pattern (each agent reads YAML directly)* — rejected: duplicates path logic ~25 times.
2. *Configuration class with caching* — rejected: complexity for no benefit (configs read once per activation).
3. *Keep bmad-init as thin shim* — rejected: PRD strategic bet says lean INTO v6.3 conventions.

**Pre-work required (FM1-2):** Audit `bmad_init.py` end-to-end before building the loader. Document every transformation it applies (env variable overrides, cascading defaults, `{project-root}` expansion, etc.). The new loader must reproduce all transformations or explicitly drop them with rationale.

**Backwards compatibility (WR8):**

```js
function loadModuleConfig(projectRoot, moduleConfigPath) {
  const configPath = path.join(projectRoot, '_bmad', moduleConfigPath, 'config.yaml');
  if (!fs.existsSync(configPath)) {
    const oldInitPath = path.join(projectRoot, '_bmad', 'core', 'bmad-init');
    if (fs.existsSync(oldInitPath)) {
      console.warn(
        `[DEPRECATED] bmad-init detected. Run 'convoke-update' to migrate. ` +
        `Support removed in Convoke 4.1.`
      );
      return loadLegacyConfig(projectRoot, moduleConfigPath);
    }
    throw new Error(`Config not found: ${configPath}`);
  }
  return yaml.parse(fs.readFileSync(configPath, 'utf8'));
}
```

Warning emitted at agent activation + `convoke-doctor` health check.

**Testability (WR7):** `tests/lib/config-loader.test.js` — v4 load, v3 fallback + warning, nested module paths, missing config error, malformed YAML actionable error, `{project-root}` expansion if applicable.

### Decision 2: Migration Pipeline Architecture (WR2)

**Decision:** Extend `migration-runner.js` + `registry.js`. New migration: `scripts/update/migrations/3.x-to-4.0.js`.

**5-phase pipeline with per-phase checkpointing:**

| Phase | Action | Checkpoint |
|-------|--------|------------|
| 1. Detect | Check for pre-v4 state (`bmad-init` exists? old activation pattern?) | `phase1_detect: complete` |
| 2. Config layout | Verify direct-load paths exist, merge if needed | `phase2_config: complete` |
| 3. Agent sweep | **Template-based rewrite** of activation section in each SKILL.md (revised per FM2-1 — NOT substring replacement) | `phase3_sweep: { migrated: [...], remaining: [...] }` |
| 4. Deprecation | Mark bmad-init as deprecated | `phase4_deprecation: complete` |
| 5. Validation | Run convoke-doctor (diff before vs after per FM2-3) | `phase5_validation: complete` |

**Phase 3 detail (FM2-1 revision):** Each agent's "On Activation" section is replaced using a **v4 activation template**, not a fragile substring match. The template is:

```markdown
## On Activation

1. **Load config** — Read configuration directly from `_bmad/{module}/config.yaml`:
   - Use `{user_name}` from config for greeting
   - Use `{communication_language}` from config for all communications
   - Store any other config variables as `{var-name}` and use appropriately
```

The migration inventory (M1) maps each agent to its module config path. Phase 3 reads the inventory, opens each SKILL.md, locates the "On Activation" section (H2 header match), replaces it with the template (parameterized by module path), and writes back.

**State tracking:** `_bmad/_config/migration-state-4.0.yaml` — per-phase status with per-file tracking in Phase 3. Schema validated on read (FM2-2). Lockfile at `_bmad/_config/.migration-lock` prevents concurrent runs.

**Idempotency:** Check state file before each phase. If complete → skip. If in-progress (Phase 3) → resume from `remaining` list. If all complete → delete state file.

**Testability (WR7):** `tests/lib/migration-3.x-to-4.0.test.js` — full migration from 3.x fixture, idempotent re-run, resume from interrupt, corrupt state handling. Fixture: snapshot of real 3.2.0 install (FM7-1).

### Decision 3: Governance Registry Architecture (WR3)

**Decision:** New module `scripts/audit/audit-bmm-dependencies.js`.

**CSV schema (`_bmad/_config/bmm-dependencies.csv`):**

```csv
skill_name,bmm_agent,dependency_type,source_module,registered_by,registered_date
bmad-enhance-initiatives-backlog,bmad-agent-pm,capability_extension,bme,scan-tool,2026-05-01
```

**Scan heuristic (revised per FM3-1 — restricted scope):**
1. **SKILL.md frontmatter `dependencies` field** — authoritative, no false positives
2. **Workflow step file code blocks** — grep for `bmad-agent-*` patterns in step files only
3. **Documentation body is EXCLUDED** — prevents false positives from prose mentions

Priority: frontmatter > step-file code. If found via multiple methods, record once with highest-confidence source. Transitive dependencies not tracked (FM3-2 — documented limitation).

**CSV generation:** Uses existing `csv-utils.js` from Team Factory for RFC 4180 quoting (FM3-3, reuses WR6).

**Integration:** `convoke-doctor` check + `convoke-update` post-upgrade gate + manual row addition for user custom skills.

**Testability (WR7):** `tests/lib/audit-bmm-dependencies.test.js` — frontmatter detection, step-file detection, doc-body exclusion, manual entry preservation, stale entry detection.

### Decision 4: PF1 Behavioral Equivalence Validation (WR4)

**Selected approach:** Option B — Recorded Replay with LLM-as-Judge comparison.

**4-stage pipeline:**

**Stage 1 — Baseline recording** (immediately before migration, same day — FM4-4):
- 5 agents: Emma (Vortex/bme), John (BMM), Winston (BMM), Carson (CIS), Murat (TEA)
- 4 prompts per agent (revised per FM4-3 — added deep workflow prompt):
  1. Agent activation greeting + menu
  2. First capability invocation
  3. Follow-up question
  4. Multi-step workflow entry (deepest accessible without real data)
- Save to `_bmad-output/pf1-baselines/{agent}-baseline.md`

**Stage 2 — Post-migration recording:** Identical 4-prompt sequence on same 5 agents.

**Stage 3 — LLM-as-Judge comparison:**
- Standardized judge prompt (version-controlled at `scripts/audit/pf1-judge-prompt.md`)
- **3 judge runs per agent, take median** (revised per FM4-1 — robust to variance)
- Score scale: 5 (identical behavior) → 1 (fundamentally different)

**Stage 4 — Go/no-go gate:**

| Condition | Result |
|-----------|--------|
| Median ≥ 4.0 average across 5 agents AND no single agent median ≤ 2 | **PASS** |
| Average ≥ 4.0 BUT one agent median ≤ 2 | **INVESTIGATE** — fix + retest |
| Average < 4.0 | **FAIL** — investigate root cause |

**Drift threshold T = 4.0** ("equivalent behavior, minor phrasing changes").

**Sprint 1 pre-work (FM4-2):** 30-minute CLI scriptability spike — verify Claude Code supports non-interactive prompt input. Fallback: manual recording + automated judge-only pipeline.

**Judge calibration test (FM7-2):** 2 fixture pairs (identical outputs → should score ≥ 4; deliberately different → should score ≤ 2). Regression test for judge quality.

**Infrastructure:** `scripts/audit/pf1-validation-battery.js` (orchestration) + `scripts/audit/pf1-judge-prompt.md` (version-controlled prompt) + `_bmad-output/pf1-baselines/` (recordings).

**Testability (WR7):** `tests/lib/pf1-validation-battery.test.js` — judge prompt parsing, threshold evaluation, go/no-go gate logic, calibration test with known-similarity fixtures.

## Implementation Patterns & Consistency Rules

*These patterns ensure Amelia (or any dev agent) writes consistent, compatible code across all 4.0 stories. Where Convoke already has established patterns, this section codifies them. Where 4.0 introduces new patterns, this section defines them.*

### Pattern 1: Script Module Structure

All new JS modules in `scripts/` follow this structure:

```js
'use strict';

const fs = require('fs');
const path = require('path');
const yaml = require('yaml');

/**
 * [Module purpose — one sentence]
 * @module scripts/[path]/[name]
 */

// --- Public API ---

function publicFunction(projectRoot, ...args) {
  // Implementation
}

// --- Internal helpers ---

function _helperFunction() {
  // Prefixed with underscore
}

// --- Exports ---

module.exports = { publicFunction };
```

**Rules:** `'use strict'` at top. CommonJS `require()` (not ES modules). `projectRoot` as first parameter for filesystem functions. Internal helpers prefixed with `_`. Single `module.exports` at bottom. JSDoc for public functions.

**Anti-pattern:** `export default`, ES module syntax, `process.cwd()` in path construction.

### Pattern 2: CSV Generation & Parsing

All CSV operations use **RFC 4180 quoting** via shared utility:

```js
const { formatCsvRow, parseCsvRow } = require('../../_bmad/bme/_team-factory/lib/utils/csv-utils.js');
```

**Rules:** All columns quoted if they can contain commas. Header row always present. `\n` line endings. No `content.split('\n')` for CSV parsing — use csv-utils parser. No `String.includes()` substring matching for CSV lookups.

**Anti-pattern:** Ad-hoc CSV generation, substring matching (I15/I48 known fragility).

### Pattern 3: YAML Read/Write Safety

```js
const yaml = require('yaml');
const doc = yaml.parseDocument(fs.readFileSync(configPath, 'utf8'));
if (doc.errors && doc.errors.length > 0) {
  throw new Error(`YAML parse error in ${configPath}: ${doc.errors[0].message}`);
}
if (doc.warnings && doc.warnings.length > 0) {
  console.warn(`YAML warning in ${configPath}: ${doc.warnings[0].message}`);
}
const config = doc.toJSON();
```

**Rules:** Always check `doc.errors` AND `doc.warnings` after parse. Use `parseDocument()` for round-trip preservation (comments survive). Use `yaml.parse()` only for read-only access. Never `JSON.stringify()` a YAML object and write back.

**Anti-pattern:** `yaml.load()` + `yaml.dump()` round-trip that strips comments (I10/I29).

### Pattern 4: Error Handling & User Messages

**Rules:**
- Errors: red `✗` prefix + "Fix:" line with actionable instructions
- Warnings: yellow `⚠` prefix + what to do about it
- Success: green `✓` prefix
- `process.exit()` only in CLI entry points, never in library modules — throw instead
- Library modules return result objects; CLI modules format them
- Fail-soft by default for governance checks (NFR9)

**Anti-pattern:** `console.warn()` directly in library modules (breaks structured output — I41). `process.exit()` in library code.

### Pattern 5: Test Conventions

**Rules:** `node:test` framework (not Jest). `assert/strict`. Test files at `tests/lib/{module-name}.test.js`. Fixtures at `tests/fixtures/{module-name}/`. Arrange-Act-Assert structure. `it('should [behavior] when [condition]')` naming.

**Anti-pattern:** Jest globals, `expect()` API, mocking filesystem when fixture directories are more reliable.

### Pattern 6: SKILL.md v4 Activation Template

```markdown
## On Activation

1. **Load config** — Read configuration directly from `_bmad/{module}/config.yaml`:
   - Use `{user_name}` from config for greeting
   - Use `{communication_language}` from config for all communications
   - Store any other config variables as `{var-name}` and use appropriately

2. **Continue with steps below:**
   [agent-specific activation steps]
```

**Rules:** "On Activation" is H2. Step 1 is ALWAYS direct-load config. Module path parameterized per agent. NO reference to `bmad-init`.

### Pattern 7: Migration Script Structure

```js
module.exports = {
  version: '4.0.0',
  description: 'Migrate from bmad-init to direct-load config pattern',
  detect(projectRoot) { /* returns boolean */ },
  apply(projectRoot) { /* returns { success, warnings } */ }
};
```

**Rules:** Exports `{ version, description, detect, apply }`. `detect()` returns boolean. `apply()` returns result object (not void, not throws). Delta logic only. Append to `registry.js`.

### Pattern 8: CLI Verbose Mode

**Rules:** All CLI commands support `--verbose`. Normal output = user-actionable only. Verbose output prefixed with `[verbose]`.

### Enforcement Guidelines

**All AI Agents MUST:** (1) Follow module structure pattern for new JS files, (2) Use csv-utils.js for CSV, (3) Check YAML warnings after parse, (4) Return result objects from library functions, (5) Write tests in `node:test` with `assert/strict`, (6) Use v4 activation template for SKILL.md, (7) Follow migration script contract.

**Violations caught by:** ESLint, `bmad-code-review` skill (Blind Hunter + Edge Case Hunter), `convoke-doctor` health checks, test failures.
